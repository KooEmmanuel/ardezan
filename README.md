# Ardezan

> An AI-native clothing store. Upload one photo and we'll style ten complete looks on you, or sketch out a piece in Kente that doesn't exist yet and we'll render you wearing it before a tailor brings it to life.

Ardezan is a single-owner, direct-to-consumer clothing brand built around two ideas most online stores can't do:

1. **Try-On** — instead of guessing how a shirt drapes on a model who isn't you, upload a full-body photo and an AI stylist (Google Gemini 2.5) shows you ten complete outfits on your body.
2. **Design Me** — pick a fabric from a curated library (Italian linen, hand-woven Kente, Japanese denim, Italian cashmere, Ankara wax-print, more), describe the piece you want made in plain English, and the AI renders you wearing it. A human tailor then makes the real thing.

It's a real online store first — cart, checkout (Stripe), order tracking, returns, the lot — with the AI layered on top, not bolted to a flashy demo.

This repo contains the entire system: Next.js storefront, FastAPI backend, an arq worker for AI orchestration, an admin dashboard, and the infra that ties them together.

## What's actually in here

### For the shopper
- **Catalog** — browse 70+ seeded products across Women / Men / Bespoke / New / Dresses / Outerwear / Accessories. Real product detail pages with multiple variants, sizing, stock.
- **AI Try-On** — upload one full-body photo, get ten complete styled looks on your body, generated by Gemini 2.5 Flash Image. Each look becomes a "card" you can add to cart or refine in plain English ("make this more weekend-y").
- **Design Me** — fork off the try-on flow with a mode toggle. Pick a fabric, pick a piece type (shirt, blazer, trouser, dress, coat...), describe what you want in your own words. Get a render of you wearing it plus a locked-in price estimate (fabric × yardage + tailoring). After checkout it goes to the tailor queue.
- **Bespoke landing** at `/catalog?cat=bespoke` — six curated fabric × piece pairings rendered as a showcase, each tile deep-links into Design Me with the form already filled.
- **Cart, checkout, orders** — Stripe Payment Element, guest and signed-in checkout, mixed catalog + custom-design carts (each kind handles inventory differently — custom designs are made-to-order, no hold).
- **Customer activity hub** at `/account/fitting-room` — your past try-ons, your custom designs, your orders, all in one view. Anonymous sessions claimed onto your account on signup/login.
- **Returns** — customer requests a return, admin marks it received, automatic refund + restock.

### For the operator (you)
- **Admin dashboard** at `/admin` — orders, products, customers, inventory, AI cost analytics, audit log.
- **Custom-design filter** on the admin orders list + a "Tailor brief" panel on each order detail showing the customer's brief, fabric snapshot, complexity, fit notes, and the rendered design.
- **Inventory ledger** — every stock movement (receive, sale, return, refund, manual adjustment) recorded with the actor + reason.
- **AI cost tracking** — every Gemini call records latency, cost estimate, prompt purpose. Kill switch + daily ceiling configurable.
- **Low-stock email digest** — daily summary of variants below their threshold.
- **Retention** — anonymous customer photos auto-delete after 15 minutes, AI renders after 24 hours, registered customer artifacts stay until they delete them.

### For the tailor
- The order's custom-design lines have full fabric and brief context attached to the admin order page so the tailor knows exactly what to cut and sew.

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Frontend | **Next.js 16** (App Router, RSC, ISR) | Fast SSR for SEO + image-heavy product pages; one framework for storefront and admin |
| Styling | **Tailwind v4** | Quick iteration, no theme files to maintain |
| Backend | **FastAPI** + Python 3.13 (`uv`) | Async, strict types via Pydantic v2, good for streaming AI responses (SSE) |
| Worker | **arq** (Redis-backed) | Try-on orchestration is multi-stage and can take 30s+; worker keeps the API free |
| Database | **MongoDB** (Motor async driver) | Flexible document model for product variants, AI sessions, retention TTLs |
| Cache / queue | **Redis** | arq's backend + rate-limit buckets |
| Payments | **Stripe** (Payment Element + webhooks) | Industry standard, handles SCA, generous testing |
| AI | **Google Gemini 2.5 Flash + Flash Image** via `google-genai` | Image generation + multimodal reasoning in one provider |
| Agent framework | **Google ADK 2.1** + MongoDB MCP server | Recommender agent reads the catalog over MCP rather than a custom function-call layer |
| Object storage | **Backblaze B2** (native API via `b2sdk`) | Generous free tier, plays nicely with signed URLs |
| Email | **SMTP** (Gmail in dev, swappable for SendGrid/Resend) | Order confirmations, ship notifications, password reset, return-requested |

## Run it locally

You'll need:
- **Python 3.13** with `uv` (`brew install uv`)
- **Node 20+** with `npm`
- **Docker** (runs MongoDB and Redis as containers)
- A **Google Gemini API key** (free at [aistudio.google.com](https://aistudio.google.com)) — without this the try-on flow won't render

### 1. Clone and install

```bash
git clone https://github.com/emmanuelnyatefe/ardezan.git
cd ardezan

# Backend deps
cd backend && uv sync && cd ..

# Frontend deps
cd frontend && npm install && cd ..
```

### 2. Set up env files

```bash
cp backend/.env.example backend/.env
cp frontend/.env.example frontend/.env.local
```

Then open `backend/.env` and fill in at minimum:

| Variable | What to put |
|---|---|
| `GEMINI_API_KEY` | Your Gemini key from AI Studio |
| `SESSION_SECRET_CUSTOMER` | `python -c "import secrets; print(secrets.token_urlsafe(48))"` |
| `SESSION_SECRET_ADMIN` | Same — generate a **fresh** one |
| `GUEST_TOKEN_SECRET` | Same — generate a **fresh** one |
| `ADMIN_BOOTSTRAP_EMAIL` | Your email — used to create the first admin |
| `ADMIN_BOOTSTRAP_PASSWORD` | Strong password for that admin |

Everything else has sensible defaults for local dev. Stripe, SMTP, B2 are all optional locally — the app degrades gracefully if you don't configure them (no payments, no emails sent, local file storage).

### 3. Start everything

```bash
python dev.py
```

That's it. `dev.py` starts MongoDB + Redis (via Docker), the FastAPI server with hot reload, the arq worker, and the Next.js dev server. It prints the URLs:

```
health   →  http://localhost:8000/api/v1/health
docs     →  http://localhost:8000/docs
storage  →  http://localhost:8000/api/v1/__debug__/storage
frontend →  http://localhost:3000
admin    →  http://localhost:3000/admin
```

Sign in to the admin at `/admin/login` with the bootstrap email/password you set.

### 4. Seed the catalog (one time)

```bash
cd backend
.venv/bin/python -m scripts.seed_fabrics       # 8 fabrics including Kente + Ankara
.venv/bin/python -m scripts.render_design_showcases  # render the Bespoke hero images
```

If you want product data, the admin has a "New product" flow under `/admin/products`. Or you can use the catalog seeder scripts under `backend/scripts/`.

## Environment variables

Full reference is in [`backend/.env.example`](backend/.env.example) and [`frontend/.env.example`](frontend/.env.example) with inline comments on every variable. The short version:

### Backend (required)
- `MONGO_URL`, `MONGO_DB` — your MongoDB connection
- `REDIS_URL` — your Redis connection
- `GEMINI_API_KEY` — Google AI Studio key
- `SESSION_SECRET_CUSTOMER`, `SESSION_SECRET_ADMIN`, `GUEST_TOKEN_SECRET` — three independent 48-byte secrets

### Backend (optional)
- `STRIPE_SECRET_KEY`, `STRIPE_PUBLISHABLE_KEY`, `STRIPE_WEBHOOK_SECRET` — payments
- `B2_KEY_ID`, `B2_APPLICATION_KEY`, `B2_BUCKET_NAME` + `STORAGE_BACKEND=b2` — cloud storage (defaults to local disk)
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USERNAME`, `SMTP_PASSWORD`, `SMTP_FROM_EMAIL` — transactional email
- `CORS_ALLOWED_ORIGINS` — set to your frontend origin in production
- `AI_KILL_SWITCH=true` — instantly disables every Gemini call
- `AI_DAILY_SPEND_CEILING_USD` — hard cap on AI spend per day

### Frontend (required)
- `NEXT_PUBLIC_API_BASE_URL` — where the browser finds the backend
- `NEXT_PUBLIC_SITE_URL` — public origin for canonical URLs + sitemap

## Deployment notes

The stack splits cleanly:

**Frontend → Vercel**
- One-click import from GitHub, root directory `frontend/`.
- Set `NEXT_PUBLIC_API_BASE_URL` and `NEXT_PUBLIC_SITE_URL` in the Vercel project settings.
- If you flip `STORAGE_BACKEND=b2` on the backend, also set `NEXT_PUBLIC_STORAGE_BACKEND=b2` so the Next image optimizer transcodes B2 URLs.

**Backend → Railway** (or Render / Fly.io)
- Deploy the `backend/` directory as a Python service.
- Add Redis as a Railway plugin (free hobby tier).
- MongoDB Atlas free tier handles the database — no need to host it yourself.
- Set every env var from the [reference above](#environment-variables).
- Worker runs as a second Railway service from the same image, command `uv run arq worker.main.WorkerSettings`.
- Point your Stripe webhook at `https://<railway-domain>/api/v1/webhooks/stripe` and put the signing secret in `STRIPE_WEBHOOK_SECRET`.

**Storage → Backblaze B2**
- Create a bucket, generate an App Key with read+write, set the env vars.
- The `scripts/migrate_local_to_b2.py` script uploads anything already on the local disk to B2 — useful when moving from local dev to a cloud bucket.

A more detailed operations + incident playbook lives in [`docs/RUNBOOK.md`](docs/RUNBOOK.md).
